## Access model

This framework is designed for a **centralised controller** that:
- SSHs into all target DB servers as `oracle` without a password (key-based)
- Uses `sudo -u grid` to execute asmdu as the grid user (ASM owner)
- Writes all output to a NAS share mounted on the controller

```
Controller (oracle) --SSH--> Target (oracle) --sudo--> grid --> asmcmd/asmdu
```

---

## Sudoers requirement on every target

This is the single most critical configuration item.
The rule must grant oracle the right to run as **grid specifically**, not as root.

```bash
# CORRECT: oracle runs asmdu scripts as grid only
oracle ALL=(grid) NOPASSWD: /tmp/asmdu_*.sh

# WRONG: escalates to root, not grid -- ASM env vars will be missing
# oracle ALL=(ALL) NOPASSWD: ALL

# WRONG: /bin/bash with NOPASSWD is effectively unrestricted root access
# oracle ALL=(root) NOPASSWD: /bin/bash
```

The playbook runs a pre-flight `id -un` check as grid before any collection
starts. If escalation lands on root instead of grid, the run fails immediately
with a clear diagnostic message rather than producing silent bad output.

## Bugs fixed from original version

1. `dg_roots` hardcoded in playbook -- moved to `group_vars/` and `host_vars/`
2. `/tmp/asmdu.sh` static path caused concurrent-run collisions -- now epoch-stamped
3. No controller-side check that `asmdu.sh` exists before copy attempt
4. `set -euo pipefail` in `run_daily.sh` aborted on non-zero Ansible RC -- fixed
5. No `loop_control` labels -- verbose output was unreadable
6. `regex_replace` double-escaped backslash -- normalised
7. `fail` mid-run replaced with `assert` in `pre_tasks` for early clear errors
8. No explicit `become_method` -- now declared at play level
9. No sudo pre-flight -- silent root escalation now caught before any collection
10. No growth tracking, trend data, or severity alerts in HTML report -- rebuilt

## Quick start

### 1. Deploy to controller

```bash
mkdir -p /opt/oracle/asmdu
cd /opt/oracle/asmdu
# Place all files from this repo here
# Also place asmdu.sh (Fred Denis) in this directory
ls -la asmdu.sh   # must exist
```

### 2. Configure inventory

```bash
vim inventory.ini
# Add hosts to [exadata_collectors] or [vmware_collectors]
# For non-standard DG names create host_vars/<hostname>.yml
```

### 3. Verify sudoers on targets

```bash
# On each target, as oracle:
sudo -u grid id -un   # must return: grid
```

### 4. Test connectivity

```bash
ansible -i inventory.ini all -m ping
ansible -i inventory.ini all -m shell -a "id -un" --become --become-user grid
```

### 5. Dry run on one host

```bash
ansible-playbook -i inventory.ini collect.yml --syntax-check
ansible-playbook -i inventory.ini collect.yml -l exa_prd -v
```

### 6. Full run and report

```bash
bash run_daily.sh
# Report: /mnt/asm-capacity/runs/<today>/report.html
```

### 7. Schedule via cron

```bash
crontab -e
# Run at 06:00 daily:
0 6 * * * /opt/oracle/asmdu/run_daily.sh >> /var/log/asmdu.log 2>&1
```

## Growth trend analysis

The report builder reads up to 30 days of historical `dg_summary.txt` files
(configurable via `--lookback N`) and computes per diskgroup:

| Metric | Method |
|--------|--------|
| Growth rate (TB/day) | OLS linear regression over available history |
| Growth rate (TB/week) | slope x 7 |
| Growth rate (TB/month) | slope x 30 |
| Week-over-week delta | latest minus 7-day-ago reading |
| Month-over-month delta | latest minus 30-day-ago reading |
| Days to full | free_tb / daily growth rate |
| Projected full date | today + days_to_full |

Minimum 2 data points required. Growth columns show -- until enough history exists.

## Alert thresholds

Edit constants at the top of `build_html_report.py`:

```python
CAPACITY_WARN_PCT  = 75.0   # % used -> yellow
CAPACITY_CRIT_PCT  = 85.0   # % used -> red
DAYS_TO_FULL_WARN  = 90     # days   -> yellow
DAYS_TO_FULL_CRIT  = 30     # days   -> red
```

## Adding a new platform group

1. Add hosts to `inventory.ini` under a new group e.g. `[baremetal_collectors]`
2. Create `group_vars/baremetal_collectors.yml` with `dg_roots` and `platform_label`
3. No playbook changes required

## Scaling notes

- Increase `forks` in `ansible.cfg` for larger fleets (50-100 for 100+ hosts)
- SSH key hygiene and consistent sudoers across estate are the main prerequisites
- NAS write bandwidth: all `delegate_to: localhost` writes land simultaneously on the controller
- Python on targets: RHEL 6 / OL 6 nodes may need `ansible_python_interpreter=/usr/bin/python3`
